## 2. 智能体管理

### 2.1 创建智能体

#### 步骤1：进入智能体管理

1. 点击侧边栏的"智能体"
2. 点击"创建智能体"按钮

#### 步骤2：填写基本信息

| 字段 | 说明 | 必填 |
|------|------|------|
| 名称 | 智能体的显示名称 | 是 |
| 描述 | 智能体的功能描述 | 否 |
| 分类 | 智能体的分类标签 | 否 |

#### 步骤3：选择模型

1. 点击"模型"下拉框
2. 选择可用的模型（如 GPT-4o、Claude-3-Opus等）
3. 调整模型参数：
   - **温度（Temperature）**: 控制输出的随机性（0-2，推荐0.7）
   - **最大Token数**: 单次响应的最大长度（推荐4096）
   - **Top-p**: 核采样参数（推荐1.0）

#### 步骤4：编写系统提示词

系统提示词定义了智能体的角色和行为：

```markdown
# 示例：客服助手

## 角色
你是ProductX的官方客服代表，负责处理客户咨询。

## 能力
- 回答产品功能问题
- 处理订单查询
- 解决常见技术问题

## 规则
- 保持友好、专业的态度
- 不确定的问题转交人工
- 不透露内部信息
```

**提示词编写技巧**：
1. 明确角色定义
2. 列出能力范围
3. 设定行为规则
4. 提供输出格式示例

#### 步骤5：绑定工具

1. 点击"工具"区域
2. 从可用工具列表中选择需要的工具
3. 工具说明：
   - `calculator`: 数学计算
   - `web_search`: 网络搜索
   - `code_executor`: 代码执行
   - `db_query`: 数据库查询
   - `file_ops`: 文件操作
   - `http_request`: HTTP请求

#### 步骤6：关联知识库

1. 点击"知识库"区域
2. 选择要关联的知识库
3. 可以关联多个知识库

#### 步骤7：保存并测试

1. 点击"保存"按钮
2. 进入"测试"标签页
3. 输入测试问题，查看智能体响应
4. 根据测试结果调整配置

### 2.2 智能体配置详解

#### 高级配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| 频率惩罚 | 降低重复内容的概率 | 0.0 |
| 存在惩罚 | 鼓励谈论新话题 | 0.0 |
| 停止词 | 遇到时停止生成 | - |
| 响应格式 | text/json | text |

#### 元数据配置

可以为智能体添加自定义元数据：

```json
{
  "category": "customer_service",
  "language": "zh-CN",
  "version": "1.0",
  "author": "张三"
}
```

### 2.3 发布智能体

#### 发布前检查清单

- [ ] 系统提示词完整、清晰
- [ ] 模型选择合适
- [ ] 工具绑定正确
- [ ] 知识库关联完成
- [ ] 测试通过

#### 发布步骤

1. 在智能体详情页，点击"发布"按钮
2. 确认发布信息
3. 智能体状态变为"已发布"
4. 智能体可在市场中被发现和使用

#### 版本管理

每次修改智能体配置，系统会自动创建新版本：

- **草稿版本**: 未发布的修改
- **已发布版本**: 当前生效的版本
- **历史版本**: 可以查看和恢复

### 2.4 智能体最佳实践

#### 提示词设计

1. **明确角色**: "你是XX公司的客服代表"
2. **设定边界**: "只处理产品咨询，不处理退款"
3. **提供示例**: "当用户问价格时，回答..."
4. **使用Markdown**: 结构化提示词更易读

#### 工具选择

根据智能体职责选择合适的工具：

| 场景 | 推荐工具 |
|------|---------|
| 客服助手 | web_search, db_query |
| 数据分析师 | calculator, code_executor |
| 内容创作者 | web_search, file_ops |
| 技术支持 | code_executor, http_request |

#### 性能优化

1. **精简提示词**: 避免过长的系统提示词
2. **合理设置Token限制**: 根据预期响应长度设置
3. **选择合适的模型**: 简单任务用小模型，复杂任务用大模型

## 3. 知识库使用

### 3.1 创建知识库

#### 步骤1：进入知识库管理

1. 点击侧边栏的"知识库"
2. 点击"创建知识库"按钮

#### 步骤2：配置知识库

| 配置项 | 说明 | 推荐值 |
|--------|------|--------|
| 名称 | 知识库名称 | - |
| 描述 | 知识库用途说明 | - |
| 嵌入模型 | 用于向量化的模型 | text-embedding-3-small |
| 分块大小 | 文档分块的大小 | 512 tokens |
| 分块重叠 | 相邻分块的重叠 | 50 tokens |
| 分块策略 | 分块方式 | semantic |

#### 分块策略选择

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| `fixed` | 固定大小分块 | 通用文档 |
| `sentence` | 按句子分块 | FAQ、对话 |
| `paragraph` | 按段落分块 | 文章、报告 |
| `semantic` | 语义分块（推荐） | 技术文档、长文 |

### 3.2 上传文档

#### 支持的文档格式

| 格式 | 扩展名 | 说明 |
|------|--------|------|
| PDF | .pdf | 支持扫描件OCR |
| Word | .docx | 保留格式结构 |
| Excel | .xlsx | 表格数据提取 |
| PowerPoint | .pptx | 幻灯片内容 |
| 文本 | .txt | 纯文本处理 |
| Markdown | .md | Markdown语法 |

#### 上传步骤

1. 在知识库详情页，点击"上传文档"
2. 选择要上传的文件（支持批量上传）
3. 可选：添加元数据标签
4. 点击"开始上传"
5. 等待文档处理完成（状态变为"就绪"）

#### 文档处理状态

| 状态 | 说明 | 操作 |
|------|------|------|
| 处理中 | 正在解析和向量化 | 等待 |
| 就绪 | 处理完成，可用于检索 | 可使用 |
| 失败 | 处理出错 | 查看错误日志，重新上传 |

### 3.3 知识库检索

#### 检索测试

1. 在知识库详情页，点击"检索测试"
2. 输入测试查询
3. 查看检索结果和相关度评分
4. 调整检索参数优化结果

#### 检索模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `naive` | 纯向量相似度 | 简单问答 |
| `local` | 实体聚焦检索 | 精确事实查询 |
| `global` | 主题聚焦检索 | 概念性问题 |
| `hybrid` | 混合检索（推荐） | 综合查询 |

#### 检索参数调优

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `top_k` | 返回结果数量 | 5-10 |
| `rerank` | 是否重排序 | true |
| `filters` | 元数据过滤 | 根据需要 |

### 3.4 知识库管理

#### 文档管理

- **查看文档列表**: 查看所有上传的文档
- **查看文档详情**: 查看文档的分块内容
- **删除文档**: 删除不需要的文档
- **重新处理**: 对失败的文档重新处理

#### 知识库统计

- **文档数量**: 已上传的文档数
- **分块数量**: 文档被分成的块数
- **存储大小**: 知识库占用的存储空间
- **状态分布**: 各状态文档的数量

### 3.5 知识库最佳实践

#### 文档准备

1. **文档质量**: 确保内容准确、完整、最新
2. **格式规范**: 使用清晰的标题、段落结构
3. **避免冗余**: 删除重复内容
4. **适当大小**: 单个文档不宜过大（建议<10MB）

#### 分块策略选择

| 文档类型 | 推荐策略 | 分块大小 |
|---------|---------|---------|
| 技术文档 | semantic | 512 tokens |
| FAQ | fixed | 256 tokens |
| 长篇文章 | paragraph | 1024 tokens |
| 对话记录 | sentence | 128 tokens |

#### 检索优化

1. **使用混合检索**: 结合向量和关键词搜索
2. **启用重排序**: 提高结果相关性
3. **添加元数据过滤**: 缩小检索范围
4. **定期更新**: 保持知识库内容最新

## 5. 工作流创建

### 5.1 工作流概念

#### 什么是工作流

工作流是一系列有序的任务执行步骤，用于实现复杂的业务流程。

#### 工作流组成

- **节点（Node）**: 单个任务步骤
- **边（Edge）**: 节点之间的连接
- **变量（Variable）**: 节点间传递的数据

#### 节点类型

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| `start` | 开始节点 | 工作流起点 |
| `end` | 结束节点 | 工作流终点 |
| `llm` | LLM调用 | 文本生成、分析 |
| `condition` | 条件判断 | 流程分支 |
| `parallel` | 并行执行 | 同时处理多个任务 |
| `loop` | 循环 | 重复执行直到条件满足 |
| `http` | HTTP请求 | 调用外部API |
| `code` | 代码执行 | 自定义逻辑 |
| `human` | 人工审批 | 需要人工介入 |
| `sub_workflow` | 子工作流 | 调用其他工作流 |

### 5.2 创建工作流

#### 步骤1：进入工作流管理

1. 点击侧边栏的"工作流"
2. 点击"创建工作流"按钮

#### 步骤2：设计工作流

1. **画布操作**:
   - 从左侧拖拽节点到画布
   - 连接节点（拖拽节点边缘的连接点）
   - 双击节点编辑配置

2. **节点配置**:

**LLM节点配置**:
```json
{
  "prompt": "对以下内容进行分类: {{input}}",
  "model": "gpt-4o",
  "temperature": 0.7,
  "output_variable": "classification"
}
```

**条件节点配置**:
```json
{
  "conditions": [
    {
      "if": "classification == 'technical'",
      "then": "technical_node"
    },
    {
      "if": "classification == 'billing'",
      "then": "billing_node"
    }
  ],
  "default": "general_node"
}
```

**HTTP节点配置**:
```json
{
  "url": "https://api.example.com/data",
  "method": "POST",
  "headers": {
    "Content-Type": "application/json"
  },
  "body": {
    "query": "{{input}}"
  },
  "output_variable": "api_result"
}
```

#### 步骤3：定义变量

在工作流设置中定义输入/输出变量：

```json
{
  "input_var": {
    "type": "string",
    "description": "用户输入",
    "required": true
  },
  "output_var": {
    "type": "string",
    "description": "处理结果"
  }
}
```

#### 步骤4：测试工作流

1. 点击"测试"按钮
2. 输入测试数据
3. 查看执行结果和节点详情
4. 根据结果调整工作流

#### 步骤5：发布工作流

1. 测试通过后，点击"发布"按钮
2. 工作流状态变为"已发布"
3. 可以通过API或触发器调用

### 5.3 工作流示例

#### 示例1：客户服务流程

```yaml
workflow:
  name: "客户服务处理"
  description: "处理客户咨询的自动化流程"
  
  nodes:
    - id: "start"
      type: "start"
      next: "classify"
    
    - id: "classify"
      type: "llm"
      config:
        prompt: |
          对以下客户问题进行分类:
          问题: {{customer_query}}
          
          分类选项:
          - technical: 技术问题
          - billing: 账单问题
          - general: 一般咨询
          
          只返回分类名称。
        model: "gpt-4o"
        output_variable: "classification"
      next: "route"
    
    - id: "route"
      type: "condition"
      config:
        conditions:
          - if: "classification == 'technical'"
            then: "tech_support"
          - if: "classification == 'billing'"
            then: "billing_support"
          - else: "general_support"
    
    - id: "tech_support"
      type: "llm"
      config:
        prompt: |
          作为技术支持专家，回答以下技术问题:
          问题: {{customer_query}}
          
          提供详细的解决方案。
        model: "gpt-4o"
        output_variable: "response"
      next: "end"
    
    - id: "billing_support"
      type: "human"
      config:
        approver_role: "billing_team"
        message: "账单问题需要人工处理"
      next: "end"
    
    - id: "general_support"
      type: "llm"
      config:
        prompt: |
          回答以下一般性问题:
          问题: {{customer_query}}
        model: "gpt-4o"
        output_variable: "response"
      next: "end"
    
    - id: "end"
      type: "end"
```

#### 示例2：文档处理流程

```yaml
workflow:
  name: "文档智能处理"
  description: "自动处理和分析文档"
  
  nodes:
    - id: "start"
      type: "start"
      next: "extract"
    
    - id: "extract"
      type: "code"
      config:
        code: |
          import json
          
          # 提取文档内容
          content = input.get("document_content")
          metadata = {
              "word_count": len(content.split()),
              "char_count": len(content)
          }
          
          output = {
              "content": content,
              "metadata": metadata
          }
      next: "analyze"
    
    - id: "analyze"
      type: "llm"
      config:
        prompt: |
          分析以下文档内容:
          {{content}}
          
          提取:
          1. 主要主题
          2. 关键要点
          3. 摘要
          
          以JSON格式返回。
        model: "gpt-4o"
        output_variable: "analysis"
      next: "end"
    
    - id: "end"
      type: "end"
```

### 5.4 工作流管理

#### 执行历史

- 查看所有执行记录
- 查看每次执行的详细信息
- 查看节点执行状态和耗时

#### 版本管理

- 每次修改自动创建新版本
- 可以查看历史版本
- 可以恢复到之前的版本

#### 监控告警

- 执行失败时发送通知
- 超时告警
- 性能监控

### 5.5 工作流最佳实践

#### 设计原则

1. **单一职责**: 每个节点只做一件事
2. **错误处理**: 为关键节点添加错误处理
3. **超时控制**: 设置合理的超时时间
4. **日志记录**: 记录关键节点的执行日志

#### 性能优化

1. **并行处理**: 独立任务使用并行节点
2. **缓存结果**: 相同输入复用结果
3. **精简提示词**: 减少LLM调用时间
4. **合理超时**: 避免无限等待

#### 调试技巧

1. **使用测试模式**: 在发布前充分测试
2. **查看执行日志**: 定位问题节点
3. **变量追踪**: 查看变量在节点间的传递
4. **逐步执行**: 复杂工作流分步调试

## 7. 常见问题

### 7.1 登录问题

#### 忘记密码

**解决**: 联系管理员重置密码

#### 登录失败

**可能原因**:
1. 用户名或密码错误
2. 账号被禁用
3. 网络连接问题

**解决**:
1. 检查用户名和密码
2. 联系管理员确认账号状态
3. 检查网络连接

### 7.2 智能体问题

#### 智能体响应不准确

**可能原因**:
1. 系统提示词不清晰
2. 模型选择不合适
3. 温度参数设置不当

**解决**:
1. 优化系统提示词
2. 尝试不同的模型
3. 调整温度参数（降低温度使响应更确定）

#### 智能体响应太慢

**可能原因**:
1. 模型响应时间长
2. 网络延迟
3. 系统负载高

**解决**:
1. 选择更快的模型
2. 检查网络连接
3. 联系管理员检查系统状态

### 7.3 知识库问题

#### 文档处理失败

**可能原因**:
1. 文档格式不支持
2. 文档损坏
3. 文档过大

**解决**:
1. 检查文档格式是否支持
2. 重新下载或获取文档
3. 压缩文档或拆分成小文件

#### 检索无结果

**可能原因**:
1. 知识库为空
2. 查询与文档内容不匹配
3. 检索模式不合适

**解决**:
1. 上传相关文档
2. 调整查询表述
3. 尝试不同的检索模式

### 7.4 工作流问题

#### 工作流执行失败

**可能原因**:
1. 节点配置错误
2. 依赖服务不可用
3. 超时

**解决**:
1. 查看执行日志，定位失败节点
2. 检查依赖服务状态
3. 调整超时设置

#### 工作流执行慢

**可能原因**:
1. LLM调用耗时
2. 串行执行过多
3. 外部API响应慢

**解决**:
1. 优化提示词，减少token使用
2. 使用并行节点
3. 优化外部API调用

### 7.5 性能问题

#### 系统响应慢

**可能原因**:
1. 网络延迟
2. 服务器负载高
3. 数据库查询慢

**解决**:
1. 检查网络连接
2. 联系管理员检查服务器
3. 优化查询条件

#### 页面加载慢

**可能原因**:
1. 浏览器缓存
2. 资源文件大
3. 网络带宽不足

**解决**:
1. 清除浏览器缓存
2. 联系管理员优化资源
3. 检查网络带宽
